What Problem Does Git Solve?

The "track changes" problem, but for code

The Before Times

Without version control: project_final.py, project_final_v2.py, project_final_ACTUALLY_final.py. Sound familiar? Git eliminates this chaos by tracking every change in a systematic way.

Key Insight

Changes on your computer don't automatically appear on GitHub, and vice versa. You explicitly sync between them. This is a feature, not a bug‚Äîit prevents half-finished work from breaking things for others.

Core Concepts

The vocabulary you need to communicate with Claude Code

Repository (Repo)

A project folder tracked by Git. Contains all your files plus the complete history of every change ever made.

Commit

A save point with a description. "Fixed the login bug" or "Added new dashboard." You can always go back to any commit.

Branch

A parallel universe for your code. Work on a new feature without affecting the main version. Merge when ready.

Push

Upload your commits to GitHub. "Here's my work, now it's backed up and others can see it."

Pull

Download the latest changes from GitHub. "What did my teammates change while I was away?"

Pull Request (PR)

A proposal to merge your branch into main. "Here's my work‚Äîplease review before it goes live."

Why Branches Matter

Safe experimentation without breaking things

The Branch Mental Model

main = the production-ready code everyone relies on
your-branch = your private sandbox to experiment in

Work on your branch, break things, fix things. Only merge to main when it's ready. If your experiment fails, just delete the branch‚Äîmain is untouched.
